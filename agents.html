Physical Setup

1. Plug in HDMI dummy plug ($8 Amazon — prevents macOS graphics issues)
2. Connect to ethernet (faster + more reliable than WiFi for 24/7 ops)
3. Connect power (with UPS if you have one)

System Settings:
→ General → Sharing → Remote Login (ON) — this enables SSH
→ General → Sharing → Screen Sharing (ON) — for VNC backup access
→ Energy Saver → Prevent automatic sleeping (ON)
→ Energy Saver → Start up automatically after power failure (ON)
→ Lock Screen → Start screen saver when inactive → Never

04 Telegram Bot — Master Command Center

Your Telegram bot is the primary interface to the entire operation. No app-switching, no SSHing for status checks. Everything happens through Telegram. Full setup guide: docs/operations/TELEGRAM-AGENT-SETUP.md

Why Telegram (Not Slack, Discord, or SMS)

  • Free, unlimited API, no rate limits for personal bots
  • Inline keyboard buttons for approve/merge/reject without typing
  • Rich markdown formatting for reports and status
  • Works on phone, tablet, desktop — same interface everywhere
  • Instant push notifications with custom sounds per priority
  • Bot API is simple Node.js — deployable in 30 minutes

Notification Script (Telegram Primary)

agents/scripts/notify.sh — Sends via Telegram Bot API

#!/bin/bash
# notify.sh — Primary notification via Telegram
TITLE="$1"
MESSAGE="$2"
PRIORITY="${3:-info}"
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')

# Load env
source ~/view1-studio/agents/telegram-bot/.env

# Emoji by priority
EMOJI="ℹ️"
[ "$PRIORITY" = "warning" ] && EMOJI="⚠️"
[ "$PRIORITY" = "urgent" ] && EMOJI="🚨"
[ "$PRIORITY" = "success" ] && EMOJI="✅"
[ "$PRIORITY" = "fail" ] && EMOJI="❌"

# Send via Telegram
TEXT="$EMOJI *$TITLE*
$MESSAGE
_${TIMESTAMP}_"

curl -s -X POST "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/sendMessage" \
  -d "chat_id=$TELEGRAM_CHAT_ID" \
  -d "parse_mode=Markdown" \
  -d "text=$TEXT" > /dev/null 2>&1

# Also log locally
echo "[$TIMESTAMP] [$PRIORITY] $TITLE: $MESSAGE" >> ~/view1-studio/agents/results/notifications.log

Agent Wrapper Script

agents/scripts/run-agent.sh — Wraps every agent task with notifications

#!/bin/bash
# run-agent.sh — Run an agent task with automatic notifications
# Usage: ./run-agent.sh "agent-id" "task-file.md" [--needs-approval]

AGENT_ID="$1"
TASK_FILE="$2"
NEEDS_APPROVAL="$3"
SCRIPTS_DIR="$(cd "$(dirname "$0")" && pwd)"
PROJECT_DIR="$(dirname "$(dirname "$SCRIPTS_DIR")")"
RESULTS_DIR="$PROJECT_DIR/agents/results"

cd "$PROJECT_DIR"

# Notify start
"$SCRIPTS_DIR/notify.sh" "$AGENT_ID started" "Working on: $(head -1 "$TASK_FILE" | sed 's/^# //')" "info"

# Run the agent
START_TIME=$(date +%s)
claude --worktree -p "$(cat "$TASK_FILE")

IMPORTANT: When done, create a git commit with a descriptive message and
push to a new branch named 'agent/$AGENT_ID/$(date +%Y%m%d-%H%M)'.
Then create a PR with a summary of changes using gh pr create." \
  --allowedTools "Read,Write,Edit,Bash,Glob,Grep" \
  --output-format json > "$RESULTS_DIR/$AGENT_ID-$(date +%Y%m%d-%H%M).json" 2>&1

EXIT_CODE=$?
END_TIME=$(date +%s)
DURATION=$(( (END_TIME - START_TIME) / 60 ))

# Notify completion
if [ $EXIT_CODE -eq 0 ]; then
  "$SCRIPTS_DIR/notify.sh" \
    "$AGENT_ID DONE ✅" \
    "Completed in ${DURATION}m. PR created. $([ "$NEEDS_APPROVAL" = "--needs-approval" ] && echo 'NEEDS YOUR REVIEW.' || echo '')" \
    "$([ "$NEEDS_APPROVAL" = "--needs-approval" ] && echo "urgent" || echo "info")"
else
  "$SCRIPTS_DIR/notify.sh" \
    "$AGENT_ID FAILED ❌" \
    "Failed after ${DURATION}m. Check logs: agents/results/$AGENT_ID-*.json" \
    "urgent"
fi

Agent Monitor Dashboard

agents/scripts/agent-monitor.sh — Check all agent status at a glance

#!/bin/bash
# agent-monitor.sh — Overview of all agent sessions
echo "╔═══════════════════════════════════════════════════╗"
echo "║         VIEW1 STUDIO — AGENT STATUS               ║"
echo "║         $(date '+%Y-%m-%d %H:%M:%S')                  ║"

for session in $(tmux list-sessions -F '#{session_name}' 2>/dev/null); do
  # Check if session has an active process
  PANE_PID=$(tmux list-panes -t "$session" -F '#{pane_pid}' 2>/dev/null)
  CHILD_COUNT=$(ps --ppid "$PANE_PID" 2>/dev/null | wc -l)

  if [ "$CHILD_COUNT" -gt 1 ]; then
    STATUS="🟢 RUNNING"
  else
    STATUS="⚪ IDLE"
  fi

  printf "║  %-20s %s\n" "$session" "$STATUS"
done

echo "╠═══════════════════════════════════════════════════╣"

# Show recent notifications
echo "║  RECENT NOTIFICATIONS (last 5):"
tail -5 ~/view1-studio/agents/results/notifications.log 2>/dev/null | while read line; do
  printf "║    %s\n" "$line"
done

echo "╚═══════════════════════════════════════════════════╝"

# Show pending PRs
echo ""
echo "PENDING PRs AWAITING REVIEW:"
cd ~/view1-studio && gh pr list --state open 2>/dev/null || echo "  (none)"

Approval Workflow

When Agents Need Your Approval

  • PR Merge: Every agent creates a PR. You review + merge. Use gh pr review --approve then gh pr merge from your phone via SSH/GitHub app
  • Payment Code: Stripe-related PRs are flagged --needs-approval and trigger urgent notifications. Never auto-merge payment code
  • Content Publishing: Social media posts are drafted to content/social/ — you review before posting. The content agent does NOT post directly
  • Architecture Decisions: If an agent encounters an ambiguity not covered in SPEC.md, it should document the question in the PR description and you'll answer in comments

# View1 Studio — Agent Handbook

## Project Overview
View1 Studio is a suite of AI-powered tools for photographers. The flagship
product is PhotoSorter (AI photo sorting + client gallery + payments).

## Architecture
- **Monorepo**: Turborepo with apps/ and packages/
- **Frontend**: Next.js 14+ (App Router), React 18, TypeScript strict mode
- **Styling**: Tailwind CSS — all styles via utility classes, NO custom CSS files
- **State**: Zustand (stores in src/stores/)
- **Backend**: Supabase (Postgres, Auth, Storage, Edge Functions, RLS)
- **Image CDN**: Cloudflare Images (on-demand transforms via URL params)
- **AI**: SigLIP via @xenova/transformers in Web Worker
- **Uploads**: tus-js-client + IndexedDB for resumable chunked uploads
- **Payments**: Stripe Billing (subscriptions) + Stripe Connect Standard
- **Email**: Resend + react-email
- **Deployment**: Vercel (frontend) + Supabase Cloud (backend)

## Key Documents (READ BEFORE STARTING)
- `docs/strategy/SPEC.md` — Full product specification (v2, 788 lines)
- `docs/strategy/research.md` — AI technology research and recommendations
- `docs/strategy/plan.md` — Implementation plan with code snippets and SQL

## Coding Conventions
- TypeScript strict mode everywhere — no `any` types, no `@ts-ignore`
- File naming: kebab-case for files, PascalCase for components
- Imports: Use `@/` path alias for src/ imports
- Components: Functional components with hooks, NO class components
- Props: Define with `interface` (not `type`), named `[Component]Props`
- Exports: Named exports for components, default export only for pages
- Error handling: Always handle errors explicitly, use try/catch in async
- Supabase: Use `createBrowserClient()` in client components,
  `createServerClient()` in server components/actions
- NEVER use `console.log` in production code — use a logger utility
- All database queries go through `packages/database/` — not inline

## Design System (packages/ui/)
- Colors: Use Tailwind theme tokens, not arbitrary values
- Dark theme: Default. Colors via CSS variables in tailwind.config.ts
- Border radius: `rounded-lg` (12px) for cards, `rounded-md` (8px) for inputs
- Spacing: 4px grid (p-1 = 4px, p-2 = 8px, etc.)
- Typography: DM Sans for body, JetBrains Mono for code/data
- Animations: Framer Motion, subtle (150-300ms), ease-out

## Git Conventions
- Branch naming: `agent/[agent-id]/[feature-name]`
- Commit messages: Conventional commits (feat:, fix:, chore:, docs:)
- Always create a PR with description explaining changes
- Never push to main directly — always branch + PR

## Testing
- Use Vitest for unit tests, Playwright for E2E
- Test files: `*.test.ts` co-located with source files
- Minimum: test happy path + one error case per function

## Supabase Conventions
- RLS policies on EVERY table — no exceptions
- Use `auth.uid()` in policies, never pass user ID from client
- Naming: snake_case for tables and columns
- Timestamps: Always `TIMESTAMPTZ`, always `DEFAULT NOW()`
- Soft delete via status change, not `DELETE`

## Stripe Conventions
- Webhook handler: single endpoint at `/api/webhooks/stripe`
- ALWAYS check idempotency via `stripe_events` table
- ALWAYS check `charges_enabled` before creating PaymentIntents
- Application fees via `application_fee_amount`, not transfers
- Test mode during development — use `sk_test_` keys

## File Structure Rules
- App routes go in `apps/photo-sorter/src/app/`
- Shared UI components go in `packages/ui/src/`
- Shared auth logic goes in `packages/auth/`
- Database types/queries go in `packages/database/`
- App-specific components go in `apps/photo-sorter/src/components/`
- App-specific lib code goes in `apps/photo-sorter/src/lib/`
